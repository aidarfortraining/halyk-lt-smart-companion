# Архитектурный план — Halyk Smart Travel Companion (demo)

> Источник внешнего вида — `prototype.html` (**только вид**, контент устарел — см. §14).
> Продуктовая логика/контент — `halyk_smart_travel_spec.md` (**источник правды по контенту**).
> Рамки сборки и не-цели — `PLAN.md`. **Этот файл — архитектура реализации.**

## Context

Цель — **защитить идею на демо-дне**, а не строить продакшн (`PLAN.md`, философия «проще = лучше»).
Сейчас в репо только спеки и прототип. Прототип уже содержит весь сценарий как массив `PHASES`
(фазы → шаги → чипсы → `onDone`) и дёргает Anthropic API прямо из браузера с захардкоженными
fallback'ами, но **заканчивается на прибытии в Астану** и **контентно расходится со спекой** (§14).

Этот план переносит оркестрацию на сервер (Django + LangGraph), UI пересобирает на React (сохраняя вид),
и **достраивает полный путь** до Итогов/Flywheel.

**Согласованные решения:**
1. Фронтенд: **только Companion (V2)**. Покупка билета вне scope; стартуем с засеянным билетом.
2. Источник правды состояния: **Django ORM**. LangGraph — **stateless-оркестратор поверх БД** (без checkpointer'а LangGraph). Это и обеспечивает переживание рестарта контейнера.
3. LLM: **реальные вызовы `claude-haiku-4-5` через LangChain + per-step fallback** (демо проходит и оффлайн).
4. Объём: **полный путь** — фазы 0–2 + **Этап 8** (в поездке: трекер, напоминания, экстренный блок, сувениры) + **Этап 10** (Итоги план/факт + Flywheel).
5. **Контент — из спеки** (`halyk_smart_travel_spec.md`), прототип — только внешний вид.

## 4. Доменная модель (Django ORM — источник правды)

`backend/trips/models.py`:

- **Trip** — корень прогона (один на демо):
  `id`, `route`, `transport`, `depart_at` (чт 4 июня ~20:00), `arrive_at` (пт 5 июня ~09:00),
  `pax`, `kids` (JSON: Айша 9 / Тимур 5), `hotel_name`, `hotel_address`, `is_apartments` (bool, демо=false),
  `weather` (JSON: дождь сб +14°), `emergency` (JSON: 3 контакта экстренного блока),
  `phase` (0–4), `step_index` (int), `created_at`.
- **PlanItem** — **ровно 10 пунктов** Travel Plan (спека §7):
  `trip(FK)`, `key` (hotel/docs/insur/budget/pharma/transfer/kino/resto/airba/taxi), `order`, `icon`,
  `service`, `phase`, `state` (`locked|wait|done`), `value`, `tag` (`g|a|null`), `badge`.
  *(Market/чеклист/сувениры/экстренный блок — НЕ plan-items: это события/UI, влияют на бюджет, но не на 10 пунктов.)*
- **Message** — лента чата (история переживает рестарт):
  `trip(FK)`, `order`, `kind` (`ai|user|time|concern|sys|results`), `text`, `meta` (JSON), `created_at`.
- **BudgetLine** — сквозная смета и трекер. **Поддерживает и живой бюджет, и таблицу Итогов:**
  `trip(FK)`, `category` (Предоплата/Трансфер/Питание/Развлечения/Сувениры/Непредвиденное/…),
  `plan_amount` (план, ₸), `fact_amount` (факт, ₸ или null пока не потрачено),
  `kind` (`prepaid|variable`), `order`.

**Что НЕ хранится отдельно:** активные `chips` и флаг `await_user` **выводятся** из `steps.py` по
`(phase, step_index)` — не персистятся. Состояние графа полностью описывается
`Trip.phase` + `Trip.step_index` + статусами `PlanItem` + `BudgetLine`. После рестарта `GET /state`
восстанавливает экран ровно на текущем шаге.

## 5. Ядро: LangGraph-граф («это и есть продукт»)

Граф моделирует путь клиента как карту тревог. **Stateless**: на каждый запрос состояние грузится из
Django, граф вычисляет один ход, результат персистится обратно. Checkpointer LangGraph не используется.

### 5.1 Состояние прогона (`graph/state.py`, transient TypedDict)
`trip_id`, `phase`, `step_index`, `action` (`start|answer|advance`), `user_chip` (None если start/advance),
`context` (погода, отель, дети, апарты, emergency), `emitted` (накапливаемые: ai-сообщения, sys-notice,
concern-карточки, time-divider'ы, мутации плана, дельты/транзакции бюджета, чипсы следующего шага,
новое `phase/step`, флаг `await_user`, и — на Этапе 10 — структура `results`).

### 5.2 Декларация шагов (`graph/steps.py`)
Сценарий всех 5 фаз как серверные данные (порт структуры `PHASES` из прототипа, **но контент из спеки**).
Каждый шаг: `key`, `phase`, `concern` (мысль клиента / None), `ai_prompt`, `chips[]`
(`{label, sub, value, plan_mutation, budget_tx, badge}`), `silent` (bool), `plan_item_key`,
`fallback` (текст из спеки).

### 5.3 Узлы графа (`graph/nodes.py`, `graph/journey.py`)
Узлы именованы по этапам, чтобы граф был наглядной картой путешествия:

- `router` — по `(phase, step_index, action)` выбирает активный шаг.
- `apply_answer` — `action=answer`: применяет `plan_mutation` (пункт → `done`) и `budget_tx`
  (перенос суммы расчётное→факт / добавление транзакции) выбранного чипса.
- `advance` — двигает `step_index`; **в цикле прогоняет silent-шаги** (документы, смета, трекер-нотисы,
  входные concern-карточки фаз) — они эмитят sys-notice/concern/time-divider без ожидания клиента.
- `weather_branch` (conditional) — для transfer/kino/restaurant/market подставляет **дождевой** контент.
- `branch_housing` (conditional) — hotel vs апарты: в демо hotel → шаг `airba` пропускается.
- `tracker` (фаза 3) — применяет «транзакции» поездки (трансфер/питание/сувениры), двигает живой бюджет
  расчётное→факт; при перерасходе/экономии эмитит умные нотисы (#13/#14).
- `generate` — общий узел генерации: LLM (§7) с `ai_prompt` + контекстом; при ошибке — `fallback`.
  Эмитит ai-сообщение + чипсы следующего шага, ставит `await_user=true`.
- `results` (фаза 4) — собирает таблицу план/факт из `BudgetLine` (175 000 vs 169 500), бонусы Halyk+,
  и 3 чипса Flywheel (Бурабай/Алаколь/Шымкент).
- `persist` — пишет `emitted` в Django (Message/PlanItem/BudgetLine/Trip) в одной транзакции.

**Рёбра:** `router → apply_answer → advance → (weather_branch | branch_housing | tracker) → generate/results → persist → END`.

### 5.4 HITL (human-in-the-loop)
На границе API, не через `interrupt()` LangGraph: один вызов = «закоммить ответ текущего шага → прогнать
silent-шаги → сгенерировать следующий шаг, требующий ответа → стоп (`await_user`)». Клиент отвечает
чипсом → новый вызов. Просто и совместимо с «Django = источник правды».

## 6. API-контракт (DRF, `trips/views.py`)

Все ответы — единый снапшот для рендера фронта.

| Метод | Эндпоинт | Назначение |
|---|---|---|
| `POST` | `/api/trip/start` | Идемпотентно засеять референсный Trip и вернуть состояние + первое AI-сообщение фазы 0. |
| `GET` | `/api/trip/<id>/state` | Полный снапшот: trip, 10 plan items, messages[], budget (live + lines), emergency, phase, текущие chips, `await_user`, results (если фаза 4). Восстановление после рестарта. |
| `POST` | `/api/trip/<id>/answer` | Тело `{chip_value}`. Прогоняет граф. Возврат — обновлённый снапшот. |
| `POST` | `/api/trip/<id>/advance` | Тело `{to_phase}` (1–4). Кнопки симуляции: разблокирует пункты фазы, ставит `phase/step_index`, запускает первый шаг фазы через граф. |

Формат снапшота (JSON) зеркалит прототип + добавляет: `budget{fact, estimate, total, lines[]}` с план/факт,
`emergency[]`, `results{rows[], totals, bonuses, flywheel[]}`.

## 7. Интеграция LLM (`graph/llm.py`)

- `ChatAnthropic(model=os.environ.get("ANTHROPIC_MODEL", "claude-haiku-4-5"))` из `langchain-anthropic`, ключ из `os.environ["ANTHROPIC_API_KEY"]`. Модель вынесена в `.env` (`ANTHROPIC_MODEL`), дефолт — `claude-haiku-4-5`.
- System-промпт = travel-контекст (семья, маршрут, **ночной поезд ~13 ч, июнь, дождь сб +14°**) + `step.ai_prompt`.
- **Fallback:** обёртка `try/except` + таймаут. При ошибке/пустом ключе → `step.fallback` (текст из спеки).
  Гарантирует полный проход демо оффлайн.
- Без стриминга (не-цель): один запрос → один текст. Только русский, без markdown, 1–2 предложения.

## 10. Персистентность и переживание рестарта

- Всё состояние — в SQLite (Trip/PlanItem/Message/BudgetLine).
- `docker-compose.yml` монтирует **named volume** на `db.sqlite3` → переживает `docker compose restart` и пересоздание контейнера.
- На старте фронт зовёт `start` (идемпотентно) → `state`; экран восстанавливается на текущем `phase/step` с полной историей чата и бюджетом.

## 15. Верификация (end-to-end)

- **Оффлайн-проход (ключевое НФТ):** пустой/неверный `ANTHROPIC_API_KEY` → весь путь фаза 0→1→2→3→4 (Итоги) проходит на fallback. Без 5xx.
- **Онлайн-проход:** с ключом — AI-сообщения генерируются моделью; чипсы двигают сценарий.
- **Контент-проверка (§14):** даты июньские; документ = Алия/28 июля; поезд ~13 ч; страховка 3 тарифа.
- **Бюджет-инвариант:** на каждом шаге `факт + расчётное ≈ 175 000 ₸`; факт 89 000→93 500→105 000→**169 500**; таблица Итогов сходится 175 000 vs 169 500.

- **Рестарт:** `docker compose restart` в середине → `GET /state` восстанавливает фазу/шаг/историю/бюджет.
- **Smoke-тест (единственные тесты, см. не-цели):** pytest на `/start`→`/answer`×N→`/advance` до фазы 4, проверка `phase/step`, состояний пунктов и сходимости Итогов.

## 16. Статус реализации (as-built) — отклонения от дизайна

Трекер сборки — `TASKS.md`. Ниже — осознанные расхождения с разделами выше (приняты при сборке, «проще = лучше»). Готово: вехи 0–3 (фазы 0–1).

| Тема | Дизайн (выше) | Как сделано | Почему |
|---|---|---|---|
| Версии | langgraph 0.2.x-эра | **langgraph 1.2.4** (1.x), Django 5.2.15, React 18 (шаблон Vite давал 19 → понижен) | актуальные стабильные; `StateGraph` API совместим |
| Персистентность графа (§5.3) | узлы копят `emitted` → узел `persist` пишет в одной транзакции | узлы пишут в Django напрямую, **вся `run()` обёрнута во `transaction.atomic`** во вью | та же атомарность и «Django=правда», меньше кода; нет узла `persist` |
| `chips`/`await_user` | в снапшоте | **выводятся** из `steps.py` по `(phase, step_index)` (`active_chips_and_await`), не персистятся | ровно как обещает §4; рестарт восстанавливает экран |
| Погода (§5.3 `weather_branch`) | conditional-узел подставляет дождевой контент | **узла нет**: дождевой контент зашит в шаги, `trip.weather` подаётся в тексты | демо всегда дождь (§9); холод/жара/норма — не-цель |
| Жильё (§5.3 `branch_housing`) | conditional-узел hotel/апарты | **свёрнут в `advance`**: шаги с `requires_apartments` пропускаются в hotel-path (`airba` остаётся locked) | один механизм пропуска для всех условных шагов |
| Контекст текста | — | `graph/context.py` (`text_ctx`/`full_ctx`) рендерит `{hotel}/{sat}/{total}` в fallback'ах, чипах, silent-эмитах | плейсхолдеры не утекают буквально |
| BudgetLine (§4) | строки сметы + Итоги | **10 детальных строк спеки §3**; живой бюджет (факт=Σоплаченных, расчётное=Σплан-неоплаченных) и таблица Итогов **выводятся** из них (prepaid→«Предоплата») | один источник, инвариант 175 000 держится автоматически |
| Узел `router` (§5.3) | отдельный узел | роль router'а растворена в `advance`/`apply_answer` (читают `(phase, step_index, action)`) | линейный граф `apply_answer→advance→generate`; conditional-узлы добавятся с фазами 2–4 |
| Статика фронта | Vite → `backend/static` | Vite → **`backend/frontend_dist`** (туда смотрят `TEMPLATES.DIRS` и WhiteNoise) | разнесено с `STATIC_ROOT=staticfiles` |
| Guard `/answer` | — | ложный/повторный ответ после закрытия фазы = no-op (валидация awaiting+чипа во вью + кламп `step_index`) | устранён дрейф состояния (найдено на ревью) |
| Путь к SQLite | файл на volume | путь через env `SQLITE_PATH` (дефолт `backend/db.sqlite3`) | Docker-volume на вехе 9 без правки кода |
