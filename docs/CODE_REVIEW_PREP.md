## 1. Что это за проект (за 60 секунд)

**Halyk Smart Travel Companion** — демо-прототип AI-ассистента поездки для Halyk Bank. Идея: превратить
разовую покупку билета в **«операционный штаб» семейной поездки** — ассистент ведёт клиента от брони до
возвращения и «закрывает» каждую тревогу в нужный момент (отель → документы → страховка → бюджет →
аптечка → трансфер → развлечения → ресторан → дождевики → поезд/такси → живой трекер расходов → Итоги).

**Референсный сценарий (захардкожен):** семья Айдар / Алия / Айша (9) / Тимур (5), Алматы → Астана,
ночной поезд ~13 ч, 5–7 июня 2026, путь через **отель** (не апартаменты), **дождливая суббота** (самая
богатая логика). Ключевой дифференциатор — **проактивный алерт о просрочке удостоверения Алии (28.07 → eGov)**.

**Цель — защитить идею на демо-дне, а не построить продакшн.** Это объясняет почти все упрощения
(см. раздел 11 «Non-goals» и раздел 13 «Что бы сделал в проде»).

**Что показывает прогон:** бюджет сходится 38 000 → 169 500 при плане 175 000 (Итоги 175 000 / 169 500,
экономия 5 500 ₸ 🎯). Демо работает **и онлайн (живой `claude-haiku-4-5`), и офлайн** (fallback на каждом шаге).

## 3. Архитектура с высоты птичьего полёта

```
Браузер (React SPA)
   │  fetch /api/...               каждый ответ = ОДИН полный снапшот
   ▼
DRF views (trips/views.py)         4 эндпоинта; оборачивают run() в transaction.atomic
   │  run(trip, action, chip_value)
   ▼
LangGraph GRAPH (graph/journey.py) START → apply_answer → advance → generate → END
   │  узлы читают/пишут Django напрямую (stateless поверх БД)
   ▼
Django ORM (trips/models.py)       Trip / PlanItem / Message / BudgetLine  ← ИСТОЧНИК ПРАВДЫ
   ▲
   │  build_snapshot(trip)         сериализация полного состояния обратно во фронт
   └──────────────────────────────────────────────────────────────────────
```

**Поток одного хода (на примере «клиент выбрал чипс»):**
1. Фронт шлёт `POST /api/trip/<id>/answer {chip_value}`.
2. View валидирует, что мы ждём ответа и чипс известен (защита от дрейфа), оборачивает в транзакцию.
3. `run()` дёргает граф: `apply_answer` (эмитит ответ клиента + применяет эффекты) → `advance` (прокручивает
   silent-шаги) → `generate` (генерирует следующее AI-сообщение или закрывает фазу).
4. View собирает `build_snapshot(trip)` и отдаёт фронту.
5. Фронт **целиком заменяет** своё состояние новым снапшотом (никакого мерджа — просто и предсказуемо).

## 4. ГЛАВНЫЙ ТЕЗИС ЗАЩИТЫ: «Django = источник правды, LangGraph — stateless поверх»

Это **самое важное архитектурное решение**, и его спросят первым. Выучи формулировку:

> «Долговременное состояние живёт **только в Django ORM** (`Trip.phase` + `Trip.step_index` + состояния
> `PlanItem` + факты `BudgetLine` + лог `Message`). LangGraph работает **stateless поверх БД**: на каждый
> запрос состояние читается из Django, граф вычисляет **один ход**, результат пишется обратно.
> **Checkpointer LangGraph не используется.**»

**Почему так (аргументы):**
- **Нет двойного источника правды.** Если бы состояние графа жило в checkpointer'е LangGraph И в Django —
  пришлось бы их синхронизировать. Один источник = нет рассинхрона.
- **Переживание рестарта «бесплатно».** Состояние полностью описывается строками БД. `docker compose restart`
  посреди демо → SQLite на volume цел → следующий `GET /state` рендерит ровно тот же экран. Это **важная
  демо-фича** (Travel Plan не теряется).
- **Простота.** `RunState` (`graph/state.py`) — это `TypedDict` всего из трёх полей: `trip_id`, `action`,
  `chip_value`. Он несёт **входы одного хода**, а не состояние. Узлы сами читают/пишут Django.
- **Атомарность.** Вся `run()` обёрнута во `transaction.atomic` во view — либо ход применился целиком,
  либо никак. Это заменило задуманный в дизайне отдельный узел `persist` (см. дельту в 6.8).

**Если спросят «зачем тогда вообще LangGraph?»** — см. раздел 6.1, там честный ответ.

## 5. AI-ЧАСТЬ — детально

### 5.0 Что LLM делает и чего НЕ делает (это спросят!)

**LLM отвечает ровно за одно:** сгенерировать **текст одного ai-сообщения** — короткую реплику ассистента
«языком тревоги», привязанную к текущему шагу. Всё.

**LLM НЕ делает (принципиально):**
- ❌ **не принимает решений** — куда идти по сценарию, что разблокировать, сколько списать. Это всё
  детерминированная логика на данных (`steps.py`) + узлы графа.
- ❌ **не вызывает инструменты / tools** — никакого tool-calling, function-calling, агентности.
- ❌ **не управляет бюджетом и планом** — суммы и мутации плана зашиты в чипсы/шаги, не в ответ модели.
- ❌ **не стримит** — ответ приходит целиком (демо не требует токенового стриминга, см. non-goals).

**Вывод для защиты:** «LLM — это **слой формулировки**, а не слой логики. Сценарий, бюджет и состояние
полностью детерминированы и тестируемы; модель лишь облекает каждый шаг в человеческую фразу. Поэтому
демо **воспроизводимо** и **не ломается** при недоступности сети.»

### 5.1 Разбор `graph/llm.py` (построчно)

```python
def generate(step, trip):
    ctx = text_ctx(trip)                      # {fri, sat, sun, hotel} — погода + отель
    fallback = step.fallback.format(**ctx)    # заранее заготовленный офлайн-текст (из спеки)
    if not settings.ANTHROPIC_API_KEY or not step.ai_prompt:
        return fallback                       # нет ключа ИЛИ шаг без промпта → fallback
    try:
        from langchain_anthropic import ChatAnthropic
        model = ChatAnthropic(
            model=settings.ANTHROPIC_MODEL,   # claude-haiku-4-5
            api_key=settings.ANTHROPIC_API_KEY,
            max_tokens=200, temperature=0.7, timeout=20, max_retries=1,
        )
        system = _SYSTEM.format(route=..., transport=..., kids=_kids(trip), **ctx)
        resp = model.invoke([("system", system), ("human", step.ai_prompt.format(**ctx))])
        return _extract(resp.content) or fallback   # пустой ответ → тоже fallback
    except Exception:
        return fallback                       # ЛЮБАЯ ошибка (сеть, таймаут, лимит) → fallback
```

**Ключевые моменты для защиты:**
- **Три пути в fallback:** (1) нет ключа / нет `ai_prompt`, (2) пустой ответ модели, (3) любое исключение.
  Это гарантирует, что `generate()` **никогда не падает и всегда возвращает строку**.
- **`_extract(content)`** — контент Anthropic может быть строкой ИЛИ списком блоков (`[{type:"text",...}]`).
  Функция аккуратно достаёт текст в обоих случаях. Это защита от формата ответа SDK.
- **Импорт `ChatAnthropic` внутри функции** (а не на уровне модуля) — чтобы импорт langchain не был нужен
  в офлайн-пути и не замедлял старт; в тестах с пустым ключом модель вообще не трогается.

### 5.2 Промпт-инжиниринг: system prompt

System-промпт (`_SYSTEM` в `llm.py`) — это и есть «характер» ассистента. Разбор по строкам:

```
Ты — AI-ассистент Halyk Travel. Говоришь языком тревоги и заботы клиента, не языком продажи сервисов.
Поездка: {route}, {transport}, июнь, семья 4 человека (дети {kids}).
Погода в Астане: пятница {fri}, суббота {sat}, воскресенье {sun}. Отель: {hotel}.
Принцип: каждое сообщение отвечает на конкретную мысль-тревогу клиента — он чувствует заботу, не продажу.
Только русский язык. Без markdown, без списков. Максимум 1–2 коротких предложения.
```

**Что здесь защищать:**
- **«Язык тревоги, не продажи»** — это продуктовый тезис всего проекта. Промпт инжектит его дважды
  (в начале и в «Принципе»), чтобы модель не сваливалась в рекламный тон.
- **Контекст поездки в system** (маршрут, дети с именами/возрастами, погода по дням, отель) — модель
  знает обстоятельства и может говорить предметно («с маленьким Тимуром в дождь…»). Это даёт ту самую
  «персонализацию», которую видно в прогоне.
- **Жёсткие рамки формата** (русский, без markdown, 1–2 предложения) — чтобы ответ влезал в чат-пузырь
  и совпадал по стилю с fallback-текстами. (Замечание о валидации — см. Q&A 12, вопрос про «мусор».)
- **`human`-сообщение — это `step.ai_prompt`** конкретного шага: узкая инструкция «о чём именно сказать
  сейчас» (например, для аптечки — «в ночном поезде ~13 ч важно средство от укачивания, Appteka привезёт
  набор домой»). Так каждый шаг управляет содержанием, а system — тоном.

### 5.3 Параметры модели — почему именно такие

| Параметр | Значение | Обоснование |
|---|---|---|
| `model` | `claude-haiku-4-5` (env `ANTHROPIC_MODEL`) | дёшево + быстро; задача простая (1–2 фразы), Opus избыточен |
| `max_tokens` | 200 | ответ — максимум 1–2 предложения; жёсткий потолок = низкая латентность/стоимость |
| `temperature` | 0.7 | немного живости/вариативности фраз, но без расхождения с инструкцией |
| `timeout` | 20 c | демо не должно «висеть»; по истечении → исключение → fallback |
| `max_retries` | 1 | один ретрай на сетевую флуктуацию; больше — лишняя задержка на демо |

### 5.4 Fallback-стратегия (почему это критично для демо)

**Каждый интерактивный шаг в `steps.py` несёт поле `fallback`** — готовый текст из спеки. Если ключа нет
или сеть недоступна, `generate()` рендерит `fallback.format(**ctx)` (подставляя погоду/отель) и возвращает его.

**Зачем:** демо-день — это сцена с непредсказуемым Wi-Fi. Требование было прямым: **демо обязано работать
полностью офлайн**. Поэтому «нет сети» — это не ошибка, а **штатный режим**: сценарий проходится целиком,
просто фразы берутся заготовленные, а не сгенерированные. Тесты этим же и пользуются (см. 5.6).

**Тонкость для защиты:** fallback и LLM-ответ **взаимозаменяемы по смыслу** — оба отвечают на одну и ту же
мысль-тревогу шага. LLM добавляет живость и персонализацию, но не несёт логики, которой нет в fallback.

### 5.5 Безопасность, стоимость, латентность

- **Безопасность `.format()`:** в `.format(**ctx)` подставляются **только данные сидов** (погода, имя отеля,
  суммы бюджета) и **авторские шаблоны** разработчика. **Свободный пользовательский текст в модель/шаблоны
  не попадает** — клиент общается только чипсами, а `chip_value` валидируется против известного списка во
  view (`trip_answer`) и в узле (`n_apply_answer`). Это снимает классы инъекций (template injection, prompt
  injection через ввод). См. также Q&A 12.
- **Ключ** хранится в `.env` (gitignored), читается в `settings.py` через `python-dotenv`. В код/гит не попадает.
- **Стоимость/латентность:** haiku + 200 токенов + 1–2 предложения на шаг → копейки и доли секунды; на демо
  ~10 LLM-вызовов за весь путь.

## 6. LANGGRAPH — детально (ядро)

### 6.1 «Зачем вообще LangGraph, если граф линейный из 3 узлов?» (самый острый вопрос)

Будь готов к этому — ревьюер увидит `apply_answer → advance → generate` и спросит, не оверинжиниринг ли это.
**Честный, уверенный ответ:**

> «Во-первых, **граф — это сознательная декларация продукта**: путь клиента смоделирован как карта тревог,
> и узлы названы по смыслу хода (`apply_answer`, `advance`, `generate`), так что код читается как карта
> путешествия. Во-вторых, **граф — это позвоночник для расширения**: в дизайне (`ARCHITECTURE.md §5.3`)
> заложены conditional-узлы `weather_branch`, `branch_housing`, `tracker`, `results`. В as-built их роль
> **свёрнута в данные** (`steps.py`) и в два простых узла — потому что для демо это оказалось проще и
> тестируемее (см. дельты 6.8). Структура графа позволяет вернуть их как явные узлы без переписывания.
> В-третьих — это **демо для банка про AI-оркестрацию**; LangGraph здесь и инструмент, и сообщение:
> мы показываем, что journey — это управляемый граф, а не клубок if-ов.»

**Не защищай то, чего нет.** Если давят: «да, сейчас граф тонкий и по сути линейный конвейер; это
осознанный right-sizing под объём демо, а не финальная форма». Зрелость = честность про текущую тонкость
+ ясный план расширения.

### 6.2 Сборка графа — `graph/journey.py`

```python
def _build():
    g = StateGraph(RunState)
    g.add_node("apply_answer", n_apply_answer)
    g.add_node("advance", n_advance)
    g.add_node("generate", n_generate)
    g.add_edge(START, "apply_answer")
    g.add_edge("apply_answer", "advance")
    g.add_edge("advance", "generate")
    g.add_edge("generate", END)
    return g.compile()

GRAPH = _build()            # компилируется один раз при импорте модуля

def run(trip, action, chip_value=None):
    GRAPH.invoke({"trip_id": trip.id, "action": action, "chip_value": chip_value})
    trip.refresh_from_db()  # узлы писали в БД мимо этого объекта → перечитываем
    return trip
```

**Защищать:**
- **`GRAPH` собирается один раз** на уровне модуля (не на каждый запрос) — компиляция графа не в горячем пути.
- **`run()` — единственная точка входа** в граф. View вызывает её внутри `transaction.atomic`.
- **`trip.refresh_from_db()`** обязателен: узлы получают свои `Trip` через `Trip.objects.get(pk=...)` и
  пишут в БД, поэтому переданный в `run()` объект `trip` устаревает — перечитываем перед сериализацией.
- Все три `action` (`start`/`answer`/`advance`) идут **через один и тот же граф** — различие в поведении
  задаётся внутри узлов по полю `action`, а не разными графами.

### 6.3 `RunState` — `graph/state.py`

```python
class RunState(TypedDict):
    trip_id: int
    action: str               # "start" | "answer" | "advance"
    chip_value: Optional[str]
```

**Это transient-состояние ОДНОГО хода, не состояние сессии.** Долговременное состояние — в Django (раздел 4).
Узлы возвращают `{}` (пустой апдейт) — они не копят данные в `RunState`, а пишут прямо в ORM. Это и есть
«stateless поверх БД» на уровне кода.

> ⚠️ **Расхождение с дизайном:** в `ARCHITECTURE.md §5.1` `RunState` описан богатым (с `context`, `emitted`,
> `await_user`, `results`…). As-built он минимальный — потому что `emitted` не копится в состоянии, а сразу
> пишется в Django; `await_user`/`results` **выводятся** из БД в сериализаторе, а не персистятся. Если
> спросят про несоответствие — это сознательное упрощение в пользу «Django = правда» (см. 6.8).

### 6.4 Три узла — `graph/nodes.py` (построчно)

**`n_apply_answer(state)` — применить ответ клиента.**
```python
if state["action"] != "answer": return {}        # для start/advance — no-op
trip = Trip.objects.get(pk=state["trip_id"])
steps = PHASES.get(trip.phase, [])
if trip.step_index >= len(steps): return {}       # фаза исчерпана → нечего применять
step = steps[trip.step_index]
chip = next((c for c in step.chips if c.value == state["chip_value"]), None)
if chip is None: return {}                          # неизвестный чипс → no-op (доп. защита)
_emit(trip, "user", chip.label)                     # 1) эхо ответа клиента в ленту
if step.plan_item:                                  # 2) отметить пункт плана done
    _mark_plan(trip, step.plan_item, chip.plan_value or step.plan_item, chip.plan_tag)
_apply_budget_tx(trip, chip.budget_category, chip.budget_fact)   # 3) бюджетная транзакция
if chip.is_apartments and not trip.is_apartments:   # 4) переключить ветку жилья (если апарты)
    trip.is_apartments = True; trip.save(update_fields=["is_apartments"])
return {}
```
Защищать: узел **идемпотентно безопасен** к мусору (двойная проверка `step_index` и `chip is None`),
и применяет ровно **четыре эффекта** выбранного чипса. Никакой логики «куда дальше» здесь нет — это `advance`.

**`n_advance(state)` — продвинуть указатель шага, прокрутить silent-шаги.**
```python
trip = Trip.objects.get(pk=state["trip_id"])
steps = PHASES.get(trip.phase, [])
idx = trip.step_index + 1 if state["action"] == "answer" else trip.step_index
while idx < len(steps):
    step = steps[idx]
    if step.silent:                 # документы, смета, входные карточки фаз, трекер-нотисы
        _run_silent(trip, step); idx += 1
    elif not _step_active(trip, step):
        idx += 1                    # apartments-only шаг в hotel-path → пропустить БЕЗ эмита
    else:
        break                       # активный интерактивный шаг → стоп, ждём клиента
trip.step_index = min(idx, len(steps))   # КЛАМП: не уехать за конец фазы
trip.save(update_fields=["step_index"])
return {}
```
**Это сердце оркестрации.** Защищать тщательно:
- **Инициализация `idx` зависит от `action`:** после `answer` сдвигаемся на шаг вперёд (текущий уже применён
  в `apply_answer`); для `start`/`advance` остаёмся на текущем индексе (его шаг нужно **показать**, а не пропустить).
- **Цикл silent-шагов** — то, что создаёт «магию проактивности»: после брони отеля автоматически, без
  действия клиента, эмитятся документы (включая **eGov-алерт про Алию**) и смета бюджета. Цикл крутится,
  пока не упрётся в активный интерактивный шаг или конец фазы.
- **Пропуск неактивных шагов** (`_step_active` = False для apartments-only `airba` в hotel-path) — **без
  эмита**: шаг просто проматывается. Это as-built `branch_housing` (см. 6.8).
- **Кламп `min(idx, len(steps))`** — защита от дрейфа `step_index` за конец фазы (нашлось на ревью; см. 6.6).

**`n_generate(state)` — показать ожидающий шаг или закрыть фазу.**
```python
trip = Trip.objects.get(pk=state["trip_id"])
steps = PHASES.get(trip.phase, [])
idx = trip.step_index
if idx < len(steps):
    _emit(trip, "ai", llm.generate(steps[idx], trip))   # AI-сообщение текущего шага
else:
    closing = PHASE_END.get(trip.phase)                 # фаза исчерпана → закрывающий sys-нотис
    if closing: _emit(trip, "sys", closing, {"level": "ok"})
return {}
```
Защищать: **это единственное место, где зовётся LLM** (`llm.generate`). Если фаза исчерпана — эмитится
закрывающее уведомление (`PHASE_END`), а `await_user` станет False (выводится в сериализаторе) → фронт
покажет кнопку симуляции следующей фазы.

### 6.5 Вспомогательные функции узлов (важны для бюджета)

- **`_emit(trip, kind, text, meta)`** — создаёт `Message` со следующим `order`. Все реплики (ai/user/sys/
  time/concern) идут через неё → лента всегда упорядочена и переживает рестарт.
- **`_mark_plan(trip, key, value, tag, state=DONE)`** — переводит пункт Travel Plan в `done` с текстом/тегом.
- **`_run_silent(trip, step)`** — эмитит `emits` шага (sys/time/concern, с подстановкой `full_ctx`),
  применяет его бюджетные `tx`, отмечает `plan_item` если есть.
- **`_apply_budget_tx(trip, category, amount)`** — **аддитивная** транзакция: `fact_amount += amount`.
  Единый механизм и для предоплат (списываются один раз), и для трекера фазы 3 (категории накапливаются).
  **Сам эмитит нотис о перерасходе (#13)**, когда транзакция переводит категорию через план:
  `if before <= plan < after`. Это разбирается в разделе 8.

### 6.6 HITL (human-in-the-loop) — на границе API, а не через `interrupt()`

**Как сделано:** один вызов `run()` = «применить ответ текущего шага → прокрутить silent-шаги →
сгенерировать следующий шаг, требующий ответа → стоп». Клиент отвечает чипсом → новый вызов. Граф
**не приостанавливается** через `interrupt()` LangGraph и не возобновляется из checkpointer'а — он каждый
раз отрабатывает целиком от START до END за один ход.

**Почему так (защита):**
- `interrupt()` потребовал бы checkpointer для сохранения/возобновления состояния графа → второй источник
  правды → противоречие с тезисом раздела 4.
- Граница «ход = HTTP-запрос» естественно ложится на REST и на «Django = правда»: каждый запрос
  самодостаточен, состояние читается из БД на входе и пишется на выходе.

**Защита от дрейфа состояния (двухуровневая):**
1. **Во view (`trip_answer`):** ответ применяется, только если `await_user is True` **и** `chip_value` есть
   среди активных чипсов. Иначе — no-op (возврат текущего снапшота). Защищает от повторного/чужого ответа.
2. **В узле (`n_apply_answer`):** ещё раз проверяется `step_index < len(steps)` и `chip is None`.
3. **В `n_advance`:** кламп `min(idx, len(steps))`.

Это покрыто тестом `test_stray_answer_is_noop` (дубль ответа и неизвестный чипс не двигают состояние).

### 6.7 Декларативные шаги — `graph/steps.py` (данные, а не код)

Весь сценарий 5 фаз описан **данными**: два dataclass'а `Step` и `Chip`, собранные в списки `PHASE_0..PHASE_4`
и словарь `PHASES = {0:…, 4:…}`. Это «порт» структуры `PHASES` из прототипа на сервер, **но контент — из спеки**.

**`Step`** (ключевые поля): `key`, `phase`, `silent` (bool), `concern` (мысль клиента — для LLM),
`ai_prompt` (инструкция модели), `fallback` (офлайн-текст), `plan_item` (какой пункт закрывает),
`requires_apartments` (активен только на ветке апартов), `chips[]`; для silent-шагов — `emits[]`
(sys/time/concern), `tx[]` (бюджетные транзакции), `plan_value`/`plan_tag`.

**`Chip`** (ключевые поля): `label`, `value`, `sub`, `plan_value`/`plan_tag` (что записать в план),
`budget_category` + `budget_fact` (сколько списать в факт), `is_apartments`.

**`active_chips_and_await(trip)`** — функция, которая по `(phase, step_index)` возвращает `(chips, await_user)`.
**Эти данные НЕ персистятся** — они выводятся из `steps.py` на лету и в сериализаторе, и во view (для валидации
ответа). Один источник определения шага → нет рассинхрона между «что показали» и «что валидируем».

**Защита:** «Сценарий — это **данные**, а не разбросанные по коду условия. Добавить шаг = добавить запись в
список. Логика (узлы) не знает про конкретные шаги — она исполняет любой список `Step`. Это делает сценарий
читаемым продактом и тестируемым.»

### 6.8 As-built дельты графа (дизайн vs реальность) — будь честен

Ревьюер может сравнить `ARCHITECTURE.md §5.3` (богатый граф) с кодом (3 узла). Дельты **задокументированы**
в `ARCHITECTURE.md §16`. Знай их наизусть:

| Задумано в дизайне | Как в коде | Почему так |
|---|---|---|
| Узел `router` | роль растворена в `apply_answer`/`advance` (читают `phase`/`step_index`/`action`) | линейный граф проще; conditional-узлы вернутся с ростом |
| Conditional `weather_branch` | узла нет: дождевой контент **зашит в шаги**, `trip.weather` подаётся в тексты | демо всегда «дождливая суббота» (§9); другие погоды — non-goal |
| Conditional `branch_housing` | узла нет: шаг `airba` помечен `requires_apartments`, `advance` его **пропускает** в hotel-path; во `/advance` он же исключается из разблокировки плана | ветка одна (демо = отель); данные + пропуск проще узла |
| Узел `tracker` (фаза 3) | узла нет: **аддитивные `_apply_budget_tx`** (`Step.tx` + чипсы); перерасход #13 — вычислимо, экономия #14 — нота в `trip_wrapup` | один механизм для предоплат и накопления |
| Узел `results` (фаза 4) | `serializers.build_results(trip)` собирает таблицу из `BudgetLine` при `phase==4`; шаг `results` лишь даёт сообщение + flywheel-чипсы | данные уже финальные в БД; переживает рестарт |
| Узел `persist` | узла нет: **вся `run()` в `transaction.atomic`** во view | та же атомарность, меньше кода |
| Богатый `RunState` (с `emitted`) | минимальный TypedDict; узлы пишут в Django сразу | «Django = правда», нет дубля состояния |

**Формула ответа:** «Дизайн закладывал максимально явный граф. На сборке часть узлов оказалась чище как
**данные** (`steps.py`) или как **производные в сериализаторе** (`results`), а персистентность — как
**транзакция во view**. Все расхождения сознательны и записаны в `ARCHITECTURE.md §16`. Поведение at
наблюдаемого результата идентично задуманному — и покрыто тестами.»

## 7. Модель данных — `trips/models.py`

Четыре модели, и **все вместе** полностью описывают состояние (раздел 4):

- **`Trip`** — корень одного прогона (для демо — один Trip). Несёт референсный сценарий (маршрут, даты,
  дети JSON, отель, погода JSON, emergency JSON) + **указатель прохождения `phase` (0..4) и `step_index`**.
  `is_apartments` — флаг ветки жилья (демо = False).
- **`PlanItem`** — ровно 10 строк Travel Plan. `state` ∈ {`locked`,`wait`,`done`}, `value`/`tag`/`badge`.
  Уникальность `(trip, key)` — constraint в БД (нельзя задвоить пункт).
- **`Message`** — лента чата. `kind` ∈ {`ai`,`user`,`time`,`concern`,`sys`,`results`}, `order` (порядок),
  `meta` (JSON: уровень sys-нотиса, заголовок/подзаголовок concern-карточки). История переживает рестарт.
- **`BudgetLine`** — бюджет. `plan_amount` (план), `fact_amount` (факт, **null пока не оплачено**),
  `kind` ∈ {`prepaid`,`variable`}. Предоплаты сворачиваются в «Предоплата» в Итогах; variable — отдельные строки.

**Защищать:** модель намеренно плоская и явная. `JSONField` (kids/weather/emergency) — потому что это
read-only справочные данные сценария, реляционная нормализация для демо избыточна.

## 8. Бюджетный инвариант — самая «вкусная» механика (выучи trace)

**Два уровня:** **факт** (оплачено/зарезервировано) и **расчётное** (прогноз системы). По мере оплат деньги
**перетекают из расчётного в факт**, а **итог держится ~175 000 ₸**. В `context.budget_now()`:
- `fact = Σ fact_amount` (где не null)
- `estimate = Σ plan_amount` (где `fact_amount is null`)
- `total = fact + estimate`

Пока сумма не оплачена — она в `estimate` по плану; как оплатили — `fact_amount` проставлен, и эта же сумма
ушла из `estimate` в `fact`. **Поэтому total не скачет** (если факт == плану).

**Полный trace факта (держи в голове):**
```
Сид:            факт 38 000  (только Билеты ЖД)         · расч. 137 000 · итог 175 000
+ Отель 48 000:        86 000                            (бронь)
+ Страховка 3 000:     89 000  · расч. 86 000            ← конец фазы 0
+ Аптечка 4 500:       93 500
  Трансфер:            (расчётное — реальный вызов в поездке, факт НЕ меняется)
+ Думан 8 000:        101 500
  Ресторан:           (расчётное — оплата на месте)
+ Дождевики 3 500:    105 000  · расч. 70 000            ← конец фазы 1
  Такси с вокзала:    (вызвано в фазе 2, спишется в фазе 3)
── фаза 3, живой трекер (аддитивные транзакции) ──
+ Трансфер 3 500 (вокзал) → 108 500
+ Питание 12 200 (пт)    → 120 700
+ Питание 14 000 (сб)    → 134 700
+ Непредвид. 4 500       → 139 200   ← здесь ждём ответа (duman_taxi)
+ Трансфер 4 000 (Думан) → 143 200
+ Сувениры 10 300        → 153 500   (#13: Сувениры 10 300 > план 10 000)
+ Питание 12 000 (вс)    → 165 500   (#13: Питание 38 200 > план 36 000)
+ Трансфер 4 000 (вокзал)→ 169 500   · расч. 0           ← конец фазы 3
```
**Итоговые факты категорий:** Трансфер 11 500 (план 14 000, −2 500), Питание 38 200 (план 36 000, +2 200),
Сувениры 10 300 (план 10 000, +300), Непредвиденное 4 500 (план 10 000, −5 500), Предоплата 105 000 (=план).
**Итог: план 175 000 / факт 169 500 / Δ −5 500 🎯.**

**Нотисы #13/#14:**
- **#13 (перерасход)** — **вычислимо** в `_apply_budget_tx`: эмитится, когда `before <= plan < after`.
  Срабатывает для Сувениров и Питания. Текст: «Категория … превысила план … — покрываем из буфера
  «Непредвиденное».»
- **#14 (экономия)** — **не вычисляется**, это заготовленная нота в шаге `trip_wrapup`: «Трансфер вышел
  дешевле плана на 2 500 ₸…».

**Защищать:** «Бюджет — это **не моковая картинка**, а суммирование строк `BudgetLine` в БД. Инвариант
175 000 держится механически: оплата лишь помечает строку `fact_amount`, перенося её из `estimate` в `fact`.
Перерасход обнаруживается арифметически на пересечении плана. Всё это закреплено тестом
`test_phase3_tracker_converges_to_169500` и `test_phase4_results_and_flywheel`.»

## 9. API-контракт и снапшот-архитектура

**4 эндпоинта** (`trips/urls.py`, `trips/views.py`):

| Метод | Путь | Назначение | Защита |
|---|---|---|---|
| `POST` | `/api/trip/start` | идемпотентно засеять Trip + первое AI-сообщение | `start` крутится **только если `not trip.messages.exists()`**; повторный вызов лишь вернёт состояние |
| `GET` | `/api/trip/<id>/state` | полный снапшот (восстановление после рестарта) | чистое чтение |
| `POST` | `/api/trip/<id>/answer` | `{chip_value}` → прогон графа | no-op, если не ждём ответа / чипс неизвестен |
| `POST` | `/api/trip/<id>/advance` | `{to_phase}` → разблокировать фазу, запустить её | no-op, если фаза не закрыта / `to_phase != phase+1` / вне 1..4 |

**Снапшот-архитектура (важный тезис):** **каждый** ответ API — это **один полный снапшот** для рендера:
`{trip, plan[10], messages[], budget{fact,estimate,total,lines}, emergency[], phase, chips, await_user, results?}`.
Фронт **заменяет состояние целиком** (`state/trip.tsx`, reducer `snapshot`). Поля `chips`/`await_user`/`results`
**не персистятся** — выводятся в `build_snapshot` из `steps.py` и `BudgetLine`.

**Почему так:** нет частичных апдейтов и мерджа на клиенте → нет рассинхрона UI с сервером. Один контракт,
предсказуемый рендер, тривиальное восстановление после рестарта. Цена — слать весь снапшот каждый раз;
для демо с одной лентой это незаметно.

`/advance` дополнительно при разблокировке фазы **исключает `airba`** из перевода в `wait`, если это
hotel-path (`exclude(key="airba")`) — вторая половина as-built `branch_housing` (первая — пропуск шага в `advance`).

## 12. Острые вопросы ревьюера + готовые ответы (Q&A)

**Q: Зачем LangGraph, если граф из 3 узлов линейный?**
A: См. 6.1. Граф — декларация продукта + позвоночник для conditional-узлов из дизайна; часть из них для
демо оказалась чище как данные/производные (6.8). Честно: сейчас тонкий, осознанный right-sizing.

**Q: Почему не используете checkpointer LangGraph / `interrupt()`?**
A: Чтобы не иметь второго источника правды. Django ORM — единственный источник; граф stateless; HITL на
границе API. Так состояние переживает рестарт без синхронизации двух хранилищ (раздел 4, 6.6).

**Q: Что будет, если LLM вернёт мусор / не по-русски / с markdown / промпт-инъекцию?**
A: Формат ограничен system-промптом (русский, без markdown, 1–2 предложения), модель — claude-haiku.
**Контентной валидации ответа нет** (честно: это известное ограничение, раздел 13). Но: (1) пустой/ошибочный
ответ → fallback; (2) **пользовательского свободного текста в промпт не попадает** — общение только чипсами,
значения валидируются, так что prompt injection через ввод исключён; (3) текст ответа лишь печатается в чат,
он **не управляет логикой/бюджетом**, поэтому «плохая фраза» не ломает сценарий.

**Q: Безопасность `.format()` на строках рядом с LLM?**
A: В `.format(**ctx)` идут только данные сидов + авторские шаблоны. Свободного ввода нет. Инъекций через
шаблон/ввод нет (раздел 5.5).

**Q: Двойной клик / повторная отправка ответа сдвинут состояние?**
A: Нет. Двухуровневая защита: валидация во view (`await_user` + чипс в списке) + проверки в `n_apply_answer`
+ кламп `step_index` в `n_advance`. Покрыто `test_stray_answer_is_noop` (6.6).

**Q: Конкурентные запросы по одному Trip?**
A: Демо однопользовательское, конкурентность — non-goal. `transaction.atomic` даёт атомарность одного хода;
блокировок строк нет, т.к. сценарий — последовательный диалог одного клиента.

**Q: Почему SQLite, а не Postgres?**
A: Один контейнер, один пользователь, состояние на volume переживает рестарт. Путь к файлу выносится в env
`SQLITE_PATH` — миграция на другой движок не требует правок кода логики.

**Q: Идемпотентен ли `/start`?**
A: Да. `ensure_seed()` создаёт Trip только если его нет; `run("start")` крутится только если нет сообщений.
Повторный `/start` просто вернёт текущий снапшот.

**Q: Как устроено восстановление после рестарта?**
A: Состояние полностью в БД (раздел 4). `GET /state` пересобирает снапшот из тех же таблиц; `chips`/
`await_user`/`results` выводятся из `(phase, step_index)` и `BudgetLine`. Travel Plan/лента/бюджет восстановятся 1:1.

**Q: Почему фронт заменяет состояние целиком, а не патчит?**
A: Нет мерджа → нет рассинхрона UI с сервером. Один контракт-снапшот, тривиальный рендер и восстановление
(раздел 9). Для демо цена «слать всё» незаметна.

**Q: Тесты гоняют живой LLM?**
A: Нет. Autouse-фикстура `_offline` ставит пустой `ANTHROPIC_API_KEY` → форсит fallback → тесты
**детерминированы и без сети**. Они проверяют **логику** (бюджет, переходы, план, гварды), не текст модели.

**Q: Где «проактивность» ассистента в коде?**
A: В silent-шагах и цикле `n_advance`: после ответа клиента граф **сам** прокручивает шаги, не требующие
ввода (документы с eGov-алертом, смета, входные карточки фаз, нотисы трекера) — это и есть «AI заговорил
первым в нужный момент» (6.4).

## 14. Шпаргалка: числа и факты наизусть

- **Версии:** Django 5.2.15 · DRF 3.17.1 · **langgraph 1.2.4** · langchain-anthropic 1.4.4 · React 18 · Vite.
- **Модель:** `claude-haiku-4-5`, `max_tokens=200`, `temperature=0.7`, `timeout=20`, `max_retries=1`.
- **Граф:** `START → apply_answer → advance → generate → END` (3 узла), компилируется 1 раз, stateless.
- **Состояние:** Django ORM (Trip.phase + step_index + PlanItem + BudgetLine + Message). Без checkpointer.
- **Фазы (0..4):** 0 отель→доки→страховка→бюджет · 1 аптечка→трансфер→Думан→ресторан→дождевики ·
  2 поезд→такси · 3 живой трекер/напоминания/сувениры · 4 Итоги + Flywheel.
- **Бюджет факт:** 38 000 → 86 000 → 89 000 → 93 500 → 101 500 → 105 000 → … → 169 500.
- **Итоги:** план **175 000** / факт **169 500** / Δ **−5 500**. Бонусы Halyk+ 8 500 ₸, категория повышена.
- **Дифференциатор:** silent-шаг `docs` → проактивный алерт «удостоверение Алии истекает 28.07.2026 → eGov».
- **Тесты:** backend 7/7 (pytest, офлайн-fallback), frontend 2/2 (vitest+jsdom). Браузерный прогон
  Playwright выполнен — все 5 фаз с живым LLM, все запросы 200.
- **Эндпоинты:** `POST /start` · `GET /<id>/state` · `POST /<id>/answer {chip_value}` · `POST /<id>/advance {to_phase}`.
- **3 строки защиты, если забыл всё:** (1) «Django = правда, LangGraph stateless поверх, переживает рестарт».
  (2) «LLM — слой формулировки, не логики; fallback на каждом шаге → офлайн-демо». (3) «Сценарий — это данные
  (`steps.py`), узлы исполняют любой список шагов; бюджет — суммирование `BudgetLine`, инвариант 175 000».
